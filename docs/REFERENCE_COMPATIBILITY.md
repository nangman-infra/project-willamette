## 1. Reference build (one-time)

```bash
brew install cmake                  # tool prereq

cd /tmp/bitnet-upstream

# Generate the model-specific LUT kernel header (only needed if
# building with BITNET_ARM_TL1 / BITNET_X86_TL2). For the I2_S CPU
# path we do NOT need the LUT kernels — see note below.
python3 utils/codegen_tl1.py --model bitnet_b1_58-3B \
    --BM 160,320,320 --BK 64,128,64 --bm 32,64,32

# Configure WITHOUT BITNET_ARM_TL1: the LUT path file
# (ggml-bitnet-lut.cpp) needs ~60 minutes of clang template
# instantiation in -O3 -DNDEBUG when GGML_BITNET_ARM_TL1 is set.
# Our model is quantised as I2_S which uses the MAD path
# (ggml-bitnet-mad.cpp + dequantize_row_i2_s in ggml-quants.c),
# so we skip TL1 for fast turnaround.
cmake -B build -DGGML_NATIVE=OFF -DBUILD_SHARED_LIBS=OFF

# ~7 minutes on Apple Silicon.
cmake --build build --target llama-cli llama-tokenize -j 4
```

Produces `/tmp/bitnet-upstream/build/bin/llama-cli` and
`/tmp/bitnet-upstream/build/bin/llama-tokenize`.

| Prompt | Willamette generated text | bitnet.cpp generated text | Bytes match |
| ------ | ------------------------- | ------------------------- | :---------: |
| `hello` | `" world\n    \"\"\"\n   "` | `" world\n    \"\"\"\n   "` | ✅ |

| `The capital of France is` | `" Paris. Paris is a"` | `" Paris. Paris is a"` | ✅ |
| `1 + 1 =` | `" 2\n2 "` | `" 2\n2 "` | ✅ |

The byte sequences are identical for all four prompts. Generated
token-id sequences match exactly for 3/4 prompts; for `hello`
Willamette produces 5 tokens `[1917, 198, 262, 3270, 262]` and
re-tokenisation of bitnet.cpp's generated text via `llama-tokenize`
yields 6 tokens `[1917, 198, 257, 12885, 198, 262]`. **Both encode
the same 18 bytes** (`" world\n    \"\"\"\n   "`) — this is a
BPE-segmentation ambiguity where the same byte sequence can be tokenised
multiple ways, and `llama-tokenize` re-encodes with the canonical
greedy-BPE algorithm rather than recovering the original generated
ids. Confirming token-id equality would require a patched `llama-cli`
that dumps the sampler's actual id stream; the byte match is
sufficient for Stage 5-E.

## 6. The pretokenizer fix (the real correctness finding)

### Before

`src/tokenizer/pretokenize.rs` used a **single** GPT-2 regex.

```rust
Regex::new(r"'s|'t|'re|'ve|'m|'ll|'d| ?\p{L}+| ?\p{N}+| ?[^\s\p{L}\p{N}]+|\s+")
```

For `"1 + 1 ="` this produced `[16, 489, 220, 16, 284]` =
`["1", " +", " 1", " ="]`, i.e. it MERGED space-with-punctuation into
single tokens like `" +"` (id 489) and `" ="` (id 284). The model never
sees those token ids in this kind of input — bitnet.cpp's reference
tokenizer splits them.

### Diagnostic from bitnet.cpp

`/tmp/bitnet-upstream/build/bin/llama-tokenize -p "1 + 1 ="` outputs
the warning **"missing pre-tokenizer type, using: 'default'"** and
yields `[16, 220, 10, 220, 16, 220, 28]` — `"+"`, `"="` are isolated
as standalone byte tokens.

### Root cause in pinned source

`tokenizer.ggml.pre` is missing from the GGUF, so llama.cpp's loader
(`src/llama.cpp:6391..6403` of the pinned commit) falls back to
`LLAMA_VOCAB_PRE_TYPE_DEFAULT`. The DEFAULT branch in
`src/llama-vocab.cpp:495..501` uses **three regexes applied
sequentially**, not the single GPT-2 regex:

```cpp
regex_exprs = {
    "[\\p{P}\\$\\+<=>\\^~\\|]+",                       // 1
    "'s|'t|'re|'ve|'m|'ll|'d| ?\\p{L}+| ?\\p{N}+| ?[^\\s\\p{L}\\p{N}]+|\\s+(?!\\S)",  // 2
    "\\p{N}+",                                          // 3
};
```

Each regex receives only the chunks NOT matched by earlier regexes
(see `unicode_regex_split` at `src/unicode.cpp:653`).

* Regex 1 isolates punctuation/operator runs (including `+`, `=`, `<`,
  `>`, `^`, `~`, `|`, `$`, plus everything in `\p{P}`).
* Regex 2 is the familiar GPT-2 alternation applied to the remaining
  text.
* Regex 3 splits any digit run that still sits inside a larger chunk —
  this is the reason `" 1"` becomes `[" ", "1"]` even though regex 2
  matched it as a single ` ?\p{N}+`.

### Willamette fix

`src/tokenizer/pretokenize.rs` now compiles three `regex::Regex`
instances and applies them sequentially (chunks marked as "matched"
by an earlier regex skip later regexes). Lookaround `(?!\S)` is
dropped (Rust `regex` doesn't support it) and replaced with plain
`\s+`; the semantics differ only on contexts that don't occur here
(see comments in `pretokenize.rs`).

A new unit test (`arithmetic_splits_per_default_regex`) asserts the
expected `["1", " ", "+", " ", "1", " ", "="]` chunking. All 177
existing tests still pass.

## 7. Open follow-ups

* **Top-k logit ID comparison** is not in this report because
  upstream `llama-cli` does not dump per-step logits. To run that
  comparison would require patching `llama-cli`'s sampling chain to
  log `top_k(softmax(logits))` per step (~50 line change in
  `examples/main/main.cpp` and/or `llm.sampling::print`). Worth
  doing in a follow-up Stage 5-E2 only if a future generation
  mismatch surfaces.

* **Original LLaMA 3 pre-tokenizer.** The GGUF lacks
  `tokenizer.ggml.pre`, so even bitnet.cpp falls back to DEFAULT and
  logs "GENERATION QUALITY WILL BE DEGRADED". The correct fix
  upstream would be to regenerate the GGUF with the proper
  `llama-bpe` pre-tokenizer key; both bitnet.cpp and Willamette
  inherit the same degraded behaviour for now.

* **Stage 6-C NEON** has shipped (see
  `docs/BITNET_FORWARD_PLAN.md` §13a-1). Re-running the comparison
  scripts with the NEON dispatcher active reproduces the same
  generated token IDs and the same generated text for all four
  reference prompts, so reference parity is preserved across the
  scalar→NEON transition.

* **Stage 6-B (x86 SSE2)** landed in v0.5.0 (f32 mask-add) and v0.7.0
  (int8 activation, now the x86 default). Validated on antiX
  Pentium-M 2 GHz against the scalar reference; greedy decode is
  byte-identical to the f32 SSE2 / scalar paths on the canonical
  prompt. Full numbers in [`BENCHMARKS.md`](BENCHMARKS.md).
* **Stage 6-B+ (AVX2 / AVX-512 / LUT)** remain deferred — they need
  modern x86 / SSSE3+ hosts to validate against the scalar reference.
